## 2026-05-13 — 新功能：論文圖譜（Paper Knowledge Graph）

**類型：** 功能設計 + 文件更新

### ✅ 完成事項

#### 新增文件
- `架構/KNOWLEDGE_GRAPH.md` — 論文圖譜功能完整規格書（資料模型、功能需求、序列圖、UI 設計、驗收標準）。

#### 更新文件

- `架構/ARCHITECTURE.md` — 將 `Graph Skill` 加入架構流程圖，新增 `GraphDB` 儲存層與 Skill 連線。

### 📌 功能概要

**論文圖譜** 是一套互動式圖形視覺化功能，將知識庫中的論文以引用連結圖的形式呈現。

| 層面 | 新增內容 |
|------|----------|
| **後端 Skill** | `graph_skill.py`：build_graph / expand_node / compute_metrics / generate_report / detect_communities |
| **後端 Tools** | `graph_engine.py`（NetworkX）`citation_fetcher.py`（SS API 引用擴展） |
| **API 端點** | `GET /api/graph` `POST /api/graph/expand/{id}` `POST /api/graph/add` `GET /api/graph/report` |
| **前端頁面** | `GraphPage.jsx` + `PaperGraph.jsx`（force-graph） + `NodeDetailPanel.jsx` + `GraphToolbar.jsx` |
| **新得套件** | `networkx` `python-louvain`（後端）`react-force-graph` `d3`（前端） |

**三大核心機制**
1. **力導向佈局** — 基於 D3-force + WebGL，支援千個節點以上的流暢互動。
2. **社群偵測** — Louvain 演算法自動將相似論文標色為同一群集。
3. **AI 圖譜報告** — Gemma 4 26B (gemma-4-26b-a4b-it) 分析圖結構，生成文字研究脈絡報告。

## 2026-06-03 — 第三週：功能修復與使用者體驗優化

**類型：** bugfix + UX 提升

### ✅ 完成事項

#### 1. 系統容錯與環境相依性處理
- **PDF 解析依賴捕獲**：在 `backend/agent_core.py` 中處理未安裝 `markitdown[pdf]` 導致的異常，回傳說明性提示，指引使用者安裝所需套件。
- **Gemini API 金鑰失效防護**：針對 403 憑證被判洩漏或無效金鑰異常，在 `agent_core.py` 進行補獲並導引更換 `.env`。
- **對話觸發論文摘要儲存與 URL/DOI 提取優化**：修復了當使用者輸入包含前綴文字與 URL/DOI 的對話訊息時無法直接觸發抓取的問題，改為從訊息任意位置正則提取。同時，修復了 `PaperSummary` 中 fields（如 `limitations`）為空或 `null` 時 Pydantic 拋出 `ValidationError` 的問題。在所有摘要寫入管道增加了 `any(x.title.lower() == ...)` 重複標題檢查，避免重置或重新分析時產生重複項目，確保資料確實且乾淨地儲存。

#### 2. 學術搜尋降級與速率限制處理
- **本機快取機制**：在 `backend/skills/search_skill.py` 中引進 `./data/search_cache.json` 作為歷史搜尋的本機持久化快取，當 API 遭遇 429 速率限制或超時時能自動使用快取進行模糊比對與降級回傳。
- **重試最佳化**：調整 Exponential Backoff 機制以減少請求等待時間。

#### 3. AI 人設偏離修正
- **角色邊界約束**：強化一般對話下的 `SYSTEM_PROMPT`，強制注入使用者當前的研究方向上下文，確保對話緊扣主題，避免迷失人設。

#### 4. 對話追問功能 (Suggestions)
- **建議對話生成**：後端 `agent_core.py` 於對話完成後，透過 LLM 多生成 3 個相關建議追問，並經由 `suggestions` 欄位傳回前端。
- **前端按鈕渲染**：修改 `frontend/src/pages/ChatPage.jsx` 以支援動態按鈕點選，方便用戶點選後自動發送後續問題，極大優化對話流暢度。

#### 5. 所有功能頁面跨分頁狀態連動
- **後端狀態持久快取**：在後端 `agent_core.py` 新增 `self._direction_cache`，並在 `main.py` 新增對應 GET/POST 接口，打通文獻矩陣及方向建議的後端同步。
- **前端自動拉取與儲存**：修改 `MatrixPage.jsx` 與 `DirectionPage.jsx`，於切換 Session 或重新載入時自動從後端 API 取得已生成之內容，與本地 `localStorage` 互為備份，實現跨分頁的資料一致性。

#### 6. 對話歷史與列表的後端同步（共享對話機制）
- **後端持久化儲存**：在後端 `agent_core.py` 中新增對話列表（`_conversations`）與聊天歷史紀錄（`_chat_history`）之底層儲存邏輯，並直接序列化寫入 `./data/session_data.json`。同時在 `chat` 與 `upload_paper` 方法中自動將訊息推入歷史紀錄，確保即時持久化。
- **全域 API 端點**：在 `main.py` 提供 `GET/POST /api/conversations` 與 `GET/POST /api/chat-history/{session_id}` 端點。
- **前端對接與生命週期同步**：重構前端 `App.jsx` 與 `ChatPage.jsx`，將原本基於瀏覽器 `localStorage` 的對話列表與歷史訊息拉取，改為由後端 API 取得。對話更新時自動副作用同步到後端，實作跨瀏覽器/Session 的全域對話共享。

## 技術決策記錄（ADR）

### ADR-001：選用 Gemma 4 26B (gemma-4-26b-a4b-it) 作為推理引擎
- **決策**：使用 Google Gemma 4 26B（`gemma-4-26b-a4b-it`）。
- **理由**：速度快、上下文視窗大（適合長篇論文），且對於學術摘要任務有良好表現。
- **取捨**：與 OpenAI 相比需使用 Google SDK；API 費率依使用量計算。

### ADR-002：RAG 架構選用 ChromaDB
- **決策**：本地向量儲存使用 ChromaDB。
- **理由**：輕量、無需額外服務、適合學術 demo environment，且 Python 整合簡單。
- **取捨**：不適合大規模生產環境；未來若擴展可遷移至 Pinecone 或 Weaviate。

### ADR-003：PDF 解析選用 MarkItDown
- **決策**：PDF → Markdown 轉換使用 Microsoft MarkItDown。
- **理由**：對學術論文的表格與公式保留度高，轉換後的 Markdown 適合直接注入 LLM Prompt。
- **取捨**：對掃描式 PDF（無文字層）支援有限。

### ADR-004：前端框架選用 Vite + React
- **決策**：前端使用 Vite + React（非 Next.js）。
- **理由**：專案為 SPA 無需 SSR，Vite 開發速度快，React 元件生態成熟。
- **取捨**：無 SEO 需求，SPA 已足夠。

### ADR-005：圖譜渲染選用 react-force-graph
- **決策**：前端圖形渲染使用 `react-force-graph`（基於 D3-force + WebGL）。
- **理由**：WebGL 加速在大量節點下仍保持流暢；支援 2D / 3D 切換；React 整合簡單。
- **取捨**：比純 SVG 方案檔案更大；相比 Cytoscape.js 設定彈性較低，但對本專案規模已足夠。

### ADR-006：引入 arXiv API 作為 429 降級備援
- **決策**：在 `search_skill.py` 中引入 `arXiv API` 備援。
- **理由**：免 API Key，極少遇到 429 限制，能覆蓋資工/AI 主題。
- **取捨**：對於非計算機科學/物理類文獻的覆蓋度不如 Semantic Scholar，但作為 429 Fallback 效果極佳。
